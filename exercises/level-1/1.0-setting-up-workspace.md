# Exercise 1.0: Setting Up Your Workspace

## Scenario Context

Welcome to Relay. You just joined the Core Platform team as a Product Manager. The company is a Series B project management SaaS -- 200 employees, $30M ARR, growing 40% year-over-year. Your product is used by mid-market teams (50-500 people) who manage complex, multi-project workflows.

On your first day, your CEO sends you this Slack message:

> "Hey! Excited to have you on board. Quick thought -- I've been hearing from several customers that managing dependencies between projects is a nightmare. We have all this task data and I think we could use AI to auto-detect when tasks in different projects are related. Like 'smart dependencies.' Could be a huge differentiator. Let's chat about it this week but wanted to plant the seed. Welcome aboard!"

This is the starting point for everything you'll build in this course.

---

## Template Exercise

**Deliverable:** A properly configured local workspace, ready for spec-driven development.

### Steps

1. **Fork this repository** to your own GitHub account. Click the "Fork" button in the upper right of the repo page.

2. **Clone your fork** to your local machine:
   ```
   git clone https://github.com/YOUR-USERNAME/relay-scenario.git
   cd relay-scenario
   ```

3. **Create your working directory.** This is where your completed exercises will live:
   ```
   mkdir -p my-work/level-1
   mkdir -p my-work/level-2
   mkdir -p my-work/level-3
   ```

4. **Copy the templates** you'll need for Level 1:
   ```
   cp specs/templates/problem-statement-template.md my-work/level-1/smart-dependencies-problem-statement.md
   cp specs/templates/discovery-template.md my-work/level-1/smart-dependencies-discovery.md
   cp specs/templates/spec-template.md my-work/level-1/smart-dependencies-spec.md
   ```

5. **Make your first commit:**
   ```
   git add my-work/
   git commit -m "Set up workspace for spec-driven development exercises"
   ```

6. **Read the CEO's message again.** In the next exercise, you'll turn this vague request into a falsifiable problem statement. Before you move on, jot down your initial reactions:
   - What's clear about this request?
   - What's ambiguous?
   - What questions would you ask before writing anything?

   Save your notes in `my-work/level-1/initial-reactions.md` and commit them.

## BYOP (Bring Your Own Product)

Think about a feature request you've recently received at work -- ideally one that arrived as a vague ask from a stakeholder or executive.

1. Create a file `my-work/level-1/byop-context.md`
2. Write down the original request (quote it if you can)
3. Note what's clear and what's ambiguous
4. List the questions you'd need answered before writing a problem statement

You'll use this real-world feature as your BYOP project throughout the course.
