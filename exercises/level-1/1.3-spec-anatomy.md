# Exercise 1.3: Understanding Spec Anatomy

## Scenario Context

Your discovery session surfaced great insight. You now know:
- The proof-of-concept model works at 83% precision on existing data
- No schema migration is possible before Q2 -- the solution must use the current data model
- The cross-project permissions problem is real and needs an explicit design decision
- Users want to discover dependencies *before* they're blocked, not just see a dependency graph
- The false positive rate matters a lot -- if users dismiss too many bad suggestions, they'll stop trusting the feature

You're ready to start writing the spec. But before you dive in, you need to understand what each section of the spec template does and why it's there.

---

## Template Exercise

**Deliverable:** A first draft of your Smart Dependencies spec with all sections populated (even if some sections are rough). This is a structural exercise -- you'll refine the content in later modules.

**Template:** [specs/templates/spec-template.md](../../specs/templates/spec-template.md)

### Steps

1. **Open your copy** of the spec template at `my-work/level-1/smart-dependencies-spec.md`.

2. **Fill in the header.** Set the status to "Draft" and version to "0.1" -- this is your first pass.

3. **Write the Problem Statement section.** You already wrote a detailed problem statement in Exercise 1.1. Adapt it for the spec -- this version should be more concise (3-5 sentences). The full problem statement document is your backup, not your spec section.

4. **Write the Context section.** Explain why now. Pull from your discovery findings: the CEO ask, the NPS data, the competitive pressure, and the proof-of-concept model. This section should convince a skeptical reader that this is the right problem at the right time.

5. **Stub out Requirements.** You'll refine these in Exercise 1.4, but for now, write at least:
   - 3 Must Haves (what absolutely has to ship)
   - 2 Should Haves (important but not blocking)
   - 2 Won't Haves (what you're explicitly not building)

   Don't worry about precision yet -- focus on coverage.

6. **Write the Constraints section.** Use your discovery notes. You have at least: the schema freeze, the model's precision limitations, the timeline pressure, and the permissions complexity.

7. **Draft Success Criteria.** Write at least 3 measurable criteria. Each should have a specific metric and target. Include both adoption metrics (did people use it?) and outcome metrics (did it help?).

8. **Write the UX/Design Intent section.** Describe the experience you want, not the interface. Where does the user encounter Smart Dependencies? What's the "aha" moment? What should it *feel like* to have the system flag a dependency you didn't know about?

9. **Start the Open Questions table.** Transfer the unresolved questions from your discovery document. Add any new ones that came up while writing the spec. Assign an owner to each.

10. **Commit your first draft:**
    ```
    git add my-work/level-1/smart-dependencies-spec.md
    git commit -m "First draft: Smart Dependencies spec (all sections populated)"
    ```

### Self-Check

- Does each section serve a different purpose? (If two sections say the same thing, one of them needs revision.)
- Could a new engineer joining the team read this spec and understand what you're building and why?
- Are your Won't Haves actually tempting features that someone might expect? (If they're obviously irrelevant, they're not useful as Won't Haves.)
- Does the UX/Design Intent focus on experience rather than interface elements?

## BYOP (Bring Your Own Product)

Start your real-world spec using the same template.

1. Open `my-work/level-1/byop-spec.md` (copy the template if you haven't already)
2. Populate every section with content from your BYOP problem statement and discovery
3. Mark sections where you feel uncertain with a `[TODO: need more info]` tag -- that's honest and useful
4. Pay attention to which sections feel easy and which feel hard. The hard ones are where your process has gaps.
