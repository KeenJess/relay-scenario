# Exercise 1.4: Writing Precise Requirements

## Scenario Context

You shared your first draft spec with Maya (Engineering Lead) and got this feedback:

> "The structure is good and I can follow the logic. But some of the requirements are fuzzy. 'The system should quickly detect dependencies' -- what does 'quickly' mean? And 'display relevant dependency information' -- relevant how? To whom? I need to be able to read a requirement and know, without asking you, whether it's done or not."

She's right. Vague requirements create ambiguity, and ambiguity creates rework. In this exercise, you'll sharpen every requirement until it passes the "could an engineer test this?" bar.

---

## Template Exercise

**Deliverable:** A revised Requirements section in your Smart Dependencies spec with precise, testable requirements.

**Template:** Continue working in your spec at `my-work/level-1/smart-dependencies-spec.md`.

### Steps

1. **Review your current requirements** from Exercise 1.3. For each one, apply the Precision Test:
   - Can a developer read this and know exactly when it's done?
   - Can a QA engineer write a test case from this?
   - Does it contain any of these vague words: quickly, easily, relevant, intuitive, seamlessly, user-friendly?

   If a requirement fails any of these tests, it needs revision.

2. **Rewrite your Must Haves.** You should have at least 5 Must Have requirements for Smart Dependencies. Here are the areas you need to cover (write the specific requirement for each):

   - **Detection:** How does the system detect dependencies? What data does it use? What's the minimum precision threshold?
   - **Display:** Where and how are detected dependencies shown to the user? What information is included with each suggestion?
   - **User control:** How does a user confirm or dismiss a detected dependency? What happens after they do?
   - **Scope:** Does detection happen within a project, across projects, or both? What are the boundaries?
   - **Performance:** What's the acceptable latency? How often does detection run?

   For each requirement, include a specific threshold or condition. "Detect dependencies between tasks in different projects" is vague. "Detect dependencies between tasks across all projects the user has access to, using task description similarity, assignee overlap, and temporal patterns, with a minimum precision of 80% measured against human-labeled test set" is precise.

3. **Rewrite your Should Haves.** You should have at least 3. These are important features that enhance the experience but don't block the initial release. Examples to consider:
   - Confidence scores on detected dependencies
   - Batch actions (confirm/dismiss multiple suggestions at once)
   - Notification when a confirmed dependency's upstream task status changes
   - Export or reporting on cross-project dependency chains

4. **Rewrite your Won't Haves.** You should have at least 3. These should be features a stakeholder might reasonably expect but that you've intentionally scoped out. For each, write a brief rationale in parentheses. Examples to consider:
   - Cross-workspace dependencies (why?)
   - Manual dependency creation (why?)
   - Automated task rescheduling based on dependencies (why?)
   - Dependencies involving tasks in archived projects (why?)

5. **Do a final pass with the "So What?" test.** For each requirement, ask: "If we removed this, what would go wrong?" If you can't answer that question, the requirement might not be a Must Have.

6. **Commit your revised requirements:**
   ```
   git add my-work/level-1/smart-dependencies-spec.md
   git commit -m "Sharpen requirements: all Must Haves now testable and precise"
   ```

### Self-Check

- Read each Must Have requirement out loud. Could you hand it to a developer you've never met and they'd know what to build?
- Check for "and" in your requirements. A requirement with "and" might actually be two requirements. Split them if so.
- Verify your MoSCoW priorities are honest. Is every Must Have truly required for launch? Would you delay launch if it wasn't done?
- Do your Won't Haves explain *why* each item is excluded?

## BYOP (Bring Your Own Product)

Sharpen the requirements in your real-world spec.

1. Apply the Precision Test to every requirement in `my-work/level-1/byop-spec.md`
2. Rewrite any that fail the test
3. Check your MoSCoW priorities. It's common to have too many Must Haves on your first pass -- be ruthless about what's truly required for initial value delivery
4. Add Won't Haves that you know stakeholders will ask about

**Stretch:** Share your Must Have requirements with an engineer on your team (or a technical friend) and ask: "Could you build and test this from what I've written?" Their confusion points are your revision targets.
