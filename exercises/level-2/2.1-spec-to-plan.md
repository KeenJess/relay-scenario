# Exercise 2.1: From Spec to Plan

## Scenario Context

Your Smart Dependencies spec is approved. Maya scheduled the sprint planning meeting and said: "I love that we have a real spec this time. But a spec isn't a plan. We need to break this into work items, sequence them, and figure out what we can ship in 3 two-week sprints."

Your engineering team for this feature:
- **Maya** (Engineering Lead) -- will own the backend architecture
- **Ravi** (Senior Backend Engineer) -- will build the detection engine
- **Sophie** (Frontend Engineer) -- will build the UI components
- **David** (Data Scientist) -- will productionize the ML model (part-time, shared with another team)

The constraint: David is only available 50% because he's supporting another team's feature launch. Whatever plan you create needs to account for this bottleneck.

---

## Template Exercise

**Deliverable:** A sprint plan that breaks the Smart Dependencies spec into 3 sprints of work items, with clear sequencing, dependencies between work items, and risk callouts.

### Steps

1. **Create your plan file** at `my-work/level-2/smart-dependencies-sprint-plan.md`.

2. **Decompose requirements into work items.** Go through each Must Have and Should Have requirement in your spec. For each, identify the engineering work items needed to implement it. A single requirement might need multiple work items (e.g., "detect dependencies across projects" might need: data pipeline, model integration, API endpoint, UI component).

   For each work item, note:
   - Description (what gets built)
   - Owner (Maya, Ravi, Sophie, or David)
   - Estimated effort (S/M/L -- small is < 2 days, medium is 2-5 days, large is 5+ days)
   - Dependencies (what needs to be done first)

3. **Sequence into 3 sprints.** Arrange work items into a logical sequence. Consider:
   - What needs to be done first to unblock other work? (Data pipeline before detection engine, detection engine before UI)
   - David is 50% capacity -- his work items take twice as long in calendar time
   - Sprint 1 should produce something testable, even if incomplete
   - Sprint 3 needs buffer for integration and bug fixes

4. **Define a "shippable increment" for each sprint.** What can you demo at the end of each sprint? Even Sprint 1 should produce visible progress.

   | Sprint | Duration | Shippable Increment |
   |--------|----------|-------------------|
   | 1 | 2 weeks | [e.g., Detection engine running against test data, basic results visible in dev environment] |
   | 2 | 2 weeks | [e.g., Full UI with real detection results, confirm/dismiss workflow working] |
   | 3 | 2 weeks | [e.g., Production-ready feature with performance optimization, edge cases handled, release candidate] |

5. **Identify risks and mitigations.** What could go wrong with this plan? List at least 3 risks:
   - What if David's other commitment takes more than 50% of his time?
   - What if the model's precision drops when tested on broader data?
   - What if the permissions edge case takes longer than estimated?

   For each risk, write a mitigation plan.

6. **Map plan back to spec.** Create a traceability table showing which spec requirements are covered in which sprint. Every Must Have should be in the plan. Any Should Have that didn't make the cut should be noted as "post-launch."

7. **Commit your plan:**
   ```
   git add my-work/level-2/smart-dependencies-sprint-plan.md
   git commit -m "Sprint plan: 3 sprints for Smart Dependencies"
   ```

### Self-Check

- Is every Must Have requirement from the spec covered in the plan?
- Does Sprint 1 produce something testable, or is it all invisible backend work?
- Have you accounted for David's 50% capacity constraint?
- Does the plan have buffer in Sprint 3, or are you planning to the edge?

## BYOP (Bring Your Own Product)

Create a sprint plan for your real-world feature.

1. Break your BYOP spec's requirements into work items
2. Sequence them into sprints (use your team's actual sprint cadence)
3. Identify the resource constraint or bottleneck in your real team -- there's always one
4. Map every Must Have to a sprint
