# Exercise 2.6: Capstone -- The Full Cycle

## Scenario Context

You've taken Smart Dependencies from approved spec through a full build cycle: planning, handoff, mid-cycle iteration, stakeholder management, and post-delivery assessment. You have a shipped feature and a clear picture of what worked and what didn't.

Your Engineering Manager, Alex, asks you to present a cycle retrospective at the next team offsite. "I want the other PMs to see what spec-driven development looks like end to end. Not a highlight reel -- the real story, including the messy parts."

---

## Template Exercise

**Deliverable:** A complete cycle retrospective document and a self-assessment using the Level 2 capstone rubric.

### Steps

1. **Create your cycle retrospective** at `my-work/level-2/smart-dependencies-cycle-retro.md`.

2. **Write the Cycle Timeline.** Create a narrative timeline of the full cycle:

   | Phase | Duration | What Happened | Key Decision |
   |-------|----------|--------------|-------------|
   | Spec Approval | [Date] | [Spec approved after 3 review rounds] | [Final precision threshold set at 80%] |
   | Sprint Planning | [Duration] | [3-sprint plan created, David's 50% constraint identified] | [Sequencing: data pipeline first, UI last] |
   | Sprint 1 | [2 weeks] | [What was built, what went well/poorly] | [Key decision made during Sprint 1] |
   | Sprint 2 | [2 weeks] | [Precision drop discovered, permissions edge case] | [Amended spec, revised plan] |
   | Sprint 3 | [2 weeks] | [What shipped, what didn't] | [Final scope decision] |
   | Post-Launch | [30 days] | [Adoption, outcome data, customer feedback] | [Follow-up plan] |

3. **Write the "What Spec-Driven Got Us" section.** Based on your experience across the full cycle, answer:
   - What decisions were easier because we had a spec?
   - What conversations went better?
   - Where did the spec prevent rework or misalignment?
   - Estimate: how many hours/days did the spec save vs. the time it took to write?

4. **Write the "Where We Struggled" section.** Be honest:
   - Where did the spec slow things down or feel like overhead?
   - Which spec sections were never referenced after kickoff?
   - What mid-cycle changes were hard to manage through spec amendments?
   - What would you skip next time?

5. **Write the "Process Improvements" section.** Based on the full cycle, list 3-5 specific changes you'd make to your spec-driven process:
   - Template changes (add a section? remove one? restructure?)
   - Review process changes (different reviewers? different cadence?)
   - Iteration process changes (how to handle spec amendments more smoothly?)
   - Communication changes (stakeholder updates? handoff format?)

6. **Score yourself** using the [Level 2 Capstone Rubric](../../rubrics/level-2-capstone-rubric.md). For each criterion, rate yourself 1-5 with a brief justification. Save in `my-work/level-2/capstone-self-assessment.md`.

7. **Final commit:**
   ```
   git add my-work/level-2/
   git commit -m "Level 2 Capstone: full cycle retrospective + self-assessment"
   ```

### What "Done" Looks Like

Your completed `my-work/level-2/` folder should contain:
- `smart-dependencies-sprint-plan.md` -- the sprint plan
- `smart-dependencies-handoff.md` -- the engineering handoff
- `smart-dependencies-iteration.md` -- the mid-cycle iteration decisions
- `smart-dependencies-stakeholder-comms.md` -- stakeholder communications
- `smart-dependencies-outcome-assessment.md` -- the 30-day outcome assessment
- `smart-dependencies-cycle-retro.md` -- the full cycle retrospective
- `capstone-self-assessment.md` -- rubric-based self-assessment
